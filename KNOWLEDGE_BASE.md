# Mango Helpdesk AI - Knowledge Base

ระบบบริหารจัดการสำหรับธุรกิจก่อสร้าง ครอบคลุม 14 โมดูลหลัก

## 1. Setup Master Data

### คำอธิบาย
โมดูลสำหรับตั้งค่าข้อมูลพื้นฐานของระบบ ประกอบด้วยข้อมูลบริษัท ผู้ใช้งาน หน่วยงาน รหัสสินค้า รหัสต้นทุน แผนก และข้อมูลอ้างอิงอื่นๆ ที่จำเป็นต่อการทำงานของระบบ

### ฟีเจอร์หลัก
- **ข้อมูลบริษัท**: ตั้งค่าชื่อบริษัท ที่อยู่ เลขประจำตัวผู้เสียภาษี ข้อมูลติดต่อ
- **ผู้ใช้งาน**: สร้างบัญชีผู้ใช้ กำหนดบทบาท (Admin, Manager, User, Viewer) และสิทธิ์การเข้าถึง
- **หน่วยงาน**: จัดการแผนกต่างๆ เช่น แผนกบัญชี แผนกจัดซื้อ แผนกโครงการ แผนกคลังสินค้า
- **รหัสสินค้า**: สร้างและจัดการรหัสวัสดุก่อสร้าง อุปกรณ์ เครื่องมือ
- **รหัสต้นทุน (Cost Center)**: กำหนดศูนย์ต้นทุนตามโครงการหรือแผนก
- **Chart of Accounts**: ตั้งค่าผังบัญชีตามมาตรฐานการบัญชี
- **Vendor Master**: บันทึกข้อมูลผู้ขาย ผู้รับเหมาช่วง ผู้ให้บริการ
- **Customer Master**: บันทึกข้อมูลลูกค้า เจ้าของโครงการ

### ขั้นตอนการทำงาน
1. เข้าสู่ระบบด้วยสิทธิ์ Administrator
2. ไปที่เมนู Setup > Master Data
3. เลือกหมวดข้อมูลที่ต้องการตั้งค่า (บริษัท, ผู้ใช้, หน่วยงาน, ฯลฯ)
4. กรอกข้อมูลที่จำเป็นตามฟอร์ม
5. ตรวจสอบความถูกต้องและบันทึก
6. ระบบจะสร้าง ID อัตโนมัติและเชื่อมโยงกับโมดูลอื่นๆ

### ตัวอย่างข้อมูล
- **รหัสสินค้า**: MAT-001 (ปูนซีเมนต์), MAT-002 (เหล็กเส้น), MAT-003 (ทราย)
- **Cost Center**: CC-001 (โครงการคอนโดลาดพร้าว), CC-002 (โครงการอาคารสำนักงาน)
- **Department**: DP-001 (แผนกบัญชี), DP-002 (แผนกจัดซื้อ), DP-003 (แผนกโครงการ)

### FAQ

**Q:** ต้องตั้งค่าข้อมูลหลักก่อนใช้งานระบบหรือไม่?  
**A:** ใช่ครับ การตั้งค่าข้อมูลหลักเป็นขั้นตอนแรกที่จำเป็น เพราะข้อมูลเหล่านี้จะถูกใช้อ้างอิงในทุกโมดูล หากไม่ตั้งค่า ระบบจะไม่สามารถสร้างเอกสาร บันทึกธุรกรรม หรือออกรายงานได้

**Q:** สามารถแก้ไขข้อมูลหลักหลังจากเริ่มใช้งานระบบแล้วได้หรือไม่?  
**A:** ได้ครับ แต่แนะนำให้ระมัดระวัง โดยเฉพาะข้อมูลที่มีการใช้งานในเอกสารหรือธุรกรรมแล้ว การแก้ไขอาจส่งผลกระทบต่อรายงานย้อนหลัง ควรใช้ฟังก์ชัน "Inactive" แทนการลบข้อมูล

**Q:** ระบบรองรับการตั้งค่าหลายบริษัทในฐานข้อมูลเดียวกันไหม?  
**A:** รองรับครับ ระบบสามารถจัดการข้อมูลหลายบริษัทในฐานข้อมูลเดียว (Multi-Company) โดยแต่ละบริษัทจะมีข้อมูลแยกอิสระกัน และผู้ใช้สามารถสลับบริษัทได้ตามสิทธิ์ที่กำหนด

**Q:** รหัสสินค้าและรหัสต้นทุนสามารถกำหนดเองได้หรือระบบจะสร้างอัตโนมัติ?  
**A:** สามารถเลือกได้ทั้งสองแบบครับ ระบบรองรับทั้งการกำหนดรหัสเอง (Manual Code) และการสร้างอัตโนมัติ (Auto-Generate) ตามรูปแบบที่ตั้งค่าไว้ เช่น MAT-YYYY-XXXX

## 2. Bidding (BD)

### คำอธิบาย
โมดูลสำหรับจัดการกระบวนการประมูลงานก่อสร้าง ตั้งแต่การรับข้อมูลการประมูล การเตรียมเอกสารเสนอราคา การคำนวณต้นทุน การส่งข้อเสนอ จนถึงการติดตามผลการประมูล

### ฟีเจอร์หลัก
- **บันทึกโครงการประมูล**: รายละเอียดโครงการ เจ้าของโครงการ วันเปิด-ปิดประมูล งบประมาณโครงการ
- **จัดการเอกสารประมูล**: แนบ TOR (Terms of Reference), แบบแปลน, เอกสารสัญญา
- **ประเมินราคา (Bill of Quantities - BOQ)**: คำนวณต้นทุนวัสดุ แรงงาน เครื่องจักร ค่าโสหุ้ย กำไร
- **สร้างเอกสารเสนอราคา**: สร้างใบเสนอราคาอัตโนมัติตาม Template
- **ติดตามสถานะ**: สถานะการประมูล (Draft, Submitted, Win, Lose, Cancelled)
- **วิเคราะห์ผล**: เปรียบเทียบราคาเสนอกับคู่แข่ง Win Rate Analysis

### ขั้นตอนการทำงาน
1. รับข้อมูลโครงการประมูลจากเจ้าของโครงการ
2. สร้างเอกสาร Bidding ในระบบ กรอกรายละเอียดโครงการ
3. แนบเอกสารประกอบการประมูล (TOR, แบบแปลน, ข้อกำหนด)
4. ประเมินต้นทุนโครงการใน BOQ (วัสดุ, แรงงาน, เครื่องจักร, ค่าใช้จ่ายอื่นๆ)
5. กำหนดอัตรากำไร (Markup) และคำนวณราคาเสนอ
6. สร้างเอกสารเสนอราคาและ Submit
7. อัพเดทสถานะการประมูล (Win/Lose) และเหตุผล
8. วิเคราะห์ผลการประมูลเพื่อปรับปรุงกระบวนการ

### ตัวอย่างข้อมูล
- **Project**: โครงการก่อสร้างอาคารสำนักงาน 8 ชั้น
- **Owner**: บริษัท ABC Property จำกัด
- **Bid Opening**: 15 มกราคม 2026
- **Estimated Cost**: 50,000,000 บาท
- **Our Bid**: 52,500,000 บาท (Markup 5%)
- **Status**: Win (ประกาศผลวันที่ 20 มกราคม 2026)

### FAQ

**Q:** ระบบสามารถติดตามสถานะการประมูลได้หรือไม่?  
**A:** ได้ครับ ระบบมี Dashboard แสดงสถานะการประมูลทั้งหมด สามารถกรองตามสถานะ (Draft, Submitted, Win, Lose) และแสดงสถิติ Win Rate, ยอดเสนอราคารวม, โครงการที่รอผล

**Q:** สามารถคำนวณ BOQ ในระบบได้เลยหรือต้องคำนวณภายนอก?  
**A:** คำนวณในระบบได้เลยครับ มีฟังก์ชัน BOQ Calculator รองรับการคำนวณต้นทุนแบบละเอียด สามารถแบ่งตาม Work Breakdown Structure (WBS) กำหนดราคาต่อหน่วย ปริมาณงาน และคำนวณต้นทุนรวมอัตโนมัติ

**Q:** หากแพ้ประมูล สามารถเก็บข้อมูลเพื่อวิเคราะห์ได้หรือไม่?  
**A:** ได้ครับ ระบบมีฟังก์ชัน Loss Analysis ให้บันทึกเหตุผลที่แพ้ประมูล (ราคาสูงเกินไป, ขาดประสบการณ์, เงื่อนไขไม่เหมาะสม) เพื่อนำไปปรับปรุงกลยุทธ์ในครั้งถัดไป

**Q:** สามารถสร้างเอกสารเสนอราคาแบบอัตโนมัติได้หรือไม่?  
**A:** ได้ครับ ระบบมี Template ของเอกสารเสนอราคาที่สามารถปรับแต่งได้ เมื่อคำนวณ BOQ เสร็จ ระบบจะ Generate เอกสารเสนอราคาพร้อมรายละเอียดต้นทุน เงื่อนไขการชำระเงิน และระยะเวลาดำเนินการ

## 3. Opportunity (OF)

### คำอธิบาย
โมดูลสำหรับจัดการโอกาสทางธุรกิจ (Sales Opportunity) ติดตามลูกค้าเป้าหมาย โอกาสในการขาย และแปลงโอกาสเป็นโครงการจริง

### ฟีเจอร์หลัก
- **บันทึกโอกาสทางธุรกิจ**: ชื่อโครงการ ลูกค้า งบประมาณโครงการ ความเป็นไปได้ (%)
- **Sales Pipeline**: แสดงสถานะโอกาสตามขั้นตอน (Lead, Qualified, Proposal, Negotiation, Closed Won/Lost)
- **Follow-up Activity**: บันทึกการติดตามลูกค้า การประชุม การนำเสนอ
- **Probability & Forecast**: คำนวณมูลค่าโอกาสถ่วงน้ำหนักด้วยความน่าจะเป็น
- **Conversion Tracking**: ติดตามอัตราการแปลงจาก Lead เป็น Project

### ขั้นตอนการทำงาน
1. บันทึกข้อมูล Lead จากแหล่งต่างๆ (Website, Event, Referral)
2. คุณสมบัติเบื้องต้น (Qualify) เช่น งบประมาณ Timeline Decision Maker
3. สร้าง Opportunity พร้อมประมาณการมูลค่าและความน่าจะเป็น
4. บันทึกกิจกรรมติดตาม (Call, Meeting, Proposal)
5. อัพเดทสถานะตาม Sales Stage
6. ปิดการขายเป็น Won (แปลงเป็น Project) หรือ Lost

### ตัวอย่างข้อมูล
- **Opportunity**: โครงการรีโนเวทโรงงาน
- **Customer**: บริษัท XYZ Manufacturing จำกัด
- **Estimated Value**: 30,000,000 บาท
- **Probability**: 70%
- **Weighted Value**: 21,000,000 บาท
- **Stage**: Proposal Submitted
- **Expected Close Date**: มีนาคม 2026

### FAQ

**Q:** Opportunity กับ Bidding ต่างกันอย่างไร?  
**A:** Opportunity คือโอกาสทางธุรกิจที่อยู่ในขั้นตอนการขาย (Pre-Sales) ใช้สำหรับติดตามลูกค้าเป้าหมาย ส่วน Bidding คือกระบวนการประมูลที่มีการแข่งขันกับผู้รับเหมารายอื่นและมีการเสนอราคาอย่างเป็นทางการ

**Q:** สามารถแปลง Opportunity เป็น Bidding ได้หรือไม่?  
**A:** ได้ครับ เมื่อ Opportunity เข้าสู่ขั้นตอน Proposal หรือ Negotiation สามารถ Convert เป็น Bidding เพื่อจัดการเอกสารเสนอราคาและ BOQ ได้ทันที

**Q:** ระบบคำนวณ Sales Forecast อย่างไร?  
**A:** ระบบคำนวณจาก Estimated Value × Probability % ของแต่ละ Opportunity และรวมเป็น Weighted Pipeline Value สามารถดูรายงานแยกตาม Sales Stage, Sales Person, หรือ Expected Close Date

## 4. Purchase Order (PO)

### คำอธิบาย
โมดูลสำหรับจัดการคำสั่งซื้อวัสดุ อุปกรณ์ และบริการที่จำเป็นสำหรับโครงการก่อสร้าง ครอบคลุมตั้งแต่การขอซื้อ การอนุมัติ การสั่งซื้อ จนถึงการรับสินค้า

### ฟีเจอร์หลัก
- **Purchase Requisition (PR)**: แผนกหรือโครงการขอซื้อวัสดุ
- **Request for Quotation (RFQ)**: ส่งขอใบเสนอราคาจากผู้ขายหลายราย
- **Vendor Comparison**: เปรียบเทียบราคาและเงื่อนไขจากผู้ขาย
- **Purchase Order (PO)**: สร้าง PO และส่งให้ผู้ขาย
- **Approval Workflow**: ขั้นตอนการอนุมัติตามวงเงิน
- **Goods Receipt (GR)**: บันทึกการรับสินค้าเข้าคลัง
- **PO Tracking**: ติดตามสถานะ PO (Open, Partial Received, Completed, Cancelled)

### ขั้นตอนการทำงาน
1. แผนกโครงการสร้าง Purchase Requisition (PR) ระบุรายการที่ต้องการ
2. แผนกจัดซื้อรับ PR และสร้าง RFQ ส่งให้ผู้ขายที่คัดเลือก
3. เปรียบเทียบใบเสนอราคาจากผู้ขาย (ราคา, เงื่อนไขการชำระเงิน, ระยะเวลาส่งมอบ)
4. เลือกผู้ขายและสร้าง PO
5. ส่ง PO ให้ผู้อนุมัติตามวงเงิน (เช่น >100,000 ต้องอนุมัติจาก Manager)
6. ส่ง PO ที่อนุมัติแล้วให้ผู้ขาย
7. ติดตามการจัดส่งและบันทึก Goods Receipt เมื่อรับสินค้า
8. ตรวจสอบ PO vs GR และปิด PO

### ตัวอย่างข้อมูล
- **PO Number**: PO-2026-001
- **Vendor**: บริษัท สหการช่าง จำกัด
- **Project**: โครงการคอนโดลาดพร้าว
- **Items**: ปูนซีเมนต์ 500 ถุง, เหล็กเส้น 10 ตัน, ทราย 20 คิว
- **Total Amount**: 250,000 บาท
- **Payment Terms**: Net 30
- **Delivery Date**: 25 มกราคม 2026
- **Status**: Partial Received (รับแล้ว 70%)

### FAQ

**Q:** Purchase Requisition (PR) และ Purchase Order (PO) ต่างกันอย่างไร?  
**A:** PR คือคำขอซื้อภายในองค์กร (Internal Request) ใช้สำหรับให้แผนกหรือโครงการแจ้งความต้องการวัสดุ ส่วน PO คือคำสั่งซื้ออย่างเป็นทางการที่ส่งให้ผู้ขาย (External Document) มีผลผูกพันทางกฎหมาย

**Q:** สามารถสร้าง PO หลายใบจาก PR ใบเดียวได้หรือไม่?  
**A:** ได้ครับ กรณีที่ PR มีหลายรายการและต้องการสั่งซื้อจากผู้ขายหลายราย สามารถแยกสร้าง PO ได้ตามความเหมาะสม ระบบจะ Link กลับไปที่ PR เดิม

**Q:** ระบบมีการแจ้งเตือน PO ที่ล่าช้าหรือไม่?  
**A:** มีครับ ระบบจะแจ้งเตือนอัตโนมัติเมื่อ Delivery Date ใกล้ครบหรือเกินกำหนด และยังไม่มี Goods Receipt สามารถตั้งค่าการแจ้งเตือนล่วงหน้าได้ (เช่น แจ้งเตือน 3 วันก่อน Delivery Date)

**Q:** สามารถรับสินค้าไม่ครบตาม PO ได้หรือไม่?  
**A:** ได้ครับ ระบบรองรับ Partial Goods Receipt กรณีที่ผู้ขายส่งมอบไม่ครบหรือแบ่งส่งหลายงวด สามารถบันทึก GR หลายครั้งจนครบตาม PO และระบบจะติดตามยอดคงเหลือที่ยังรอรับอัตโนมัติ

## 5. Accounts Payable (AP)

### คำอธิบาย
โมดูลสำหรับจัดการเจ้าหนี้การค้า ตั้งแต่การรับ Invoice การตรวจสอบความถูกต้อง การอนุมัติการชำระเงิน การจ่ายเงิน และการติดตามสถานะหนี้

### ฟีเจอร์หลัก
- **Invoice Recording**: บันทึก Vendor Invoice จับคู่กับ PO และ GR (3-Way Matching)
- **Payment Approval**: Workflow การอนุมัติการชำระเงินตามวงเงิน
- **Payment Schedule**: จัดการกำหนดการชำระเงินตาม Payment Terms
- **Payment Processing**: ชำระเงินผ่านโอน, เช็ค, เงินสด พร้อมบันทึกหลักฐาน
- **Aging Report**: รายงานอายุหนี้ (0-30, 31-60, 61-90, >90 วัน)

### FAQ

**Q:** 3-Way Matching คืออะไร?  
**A:** คือการตรวจสอบความถูกต้องของ Invoice โดยจับคู่กับ 3 เอกสาร ได้แก่ (1) Purchase Order (PO), (2) Goods Receipt (GR), และ (3) Invoice เพื่อให้มั่นใจว่าสิ่งที่สั่งซื้อ, รับจริง, และเรียกเก็บเงิน ตรงกัน

**Q:** ระบบคำนวณภาษีหัก ณ ที่จ่ายอัตโนมัติหรือไม่?  
**A:** คำนวณอัตโนมัติครับ ตามประเภท Vendor และประเภทค่าใช้จ่าย เช่น ค่าบริการ 3%, ค่าเช่า 5%, ค่าขนส่ง 1% ระบบจะคำนวณและหักจากยอดชำระพร้อมสร้างรายงานภาษีหัก ณ ที่จ่าย (ภ.ง.ด.53)

**Q:** สามารถชำระเงินเกินหนึ่ง Invoice ในครั้งเดียวได้หรือไม่?  
**A:** ได้ครับ ระบบรองรับการชำระเงินแบบ Batch Payment คือเลือก Invoice หลายใบของ Vendor เดียวกันมาชำระพร้อมกัน ช่วยลดจำนวนธุรกรรมและค่าธรรมเนียมการโอนเงิน

**Q:** หากชำระเงินผิดพลาดจะแก้ไขอย่างไร?  
**A:** สามารถ Reverse Payment โดยสร้าง Payment Reversal ระบบจะสร้าง Accounting Entry กลับรายการและคืนสถานะ Invoice เป็น Unpaid เพื่อทำการชำระเงินใหม่

## 6. Inventory Control (IC)

### คำอธิบาย
โมดูลสำหรับจัดการสินค้าคงคลัง วัสดุก่อสร้าง อุปกรณ์ เครื่องมือ ตั้งแต่การรับเข้า การเก็บ การเบิกจ่าย จนถึงการตรวจนับสต๊อก

### ฟีเจอร์หลัก
- **Goods Receipt**: บันทึกการรับสินค้าเข้าคลังจาก PO หรือการผลิต
- **Goods Issue**: เบิกจ่ายวัสดุให้โครงการหรือแผนก
- **Stock Transfer**: โอนย้ายสินค้าระหว่างคลัง หรือ Location
- **Stock Adjustment**: ปรับปรุงสต๊อกกรณีชำรุด สูญหาย หรือตรวจนับไม่ตรง
- **Physical Inventory**: ตรวจนับสต๊อกประจำงวด
- **Reorder Point**: แจ้งเตือนเมื่อสต๊อกต่ำกว่าจุดสั่งซื้อ
- **ABC Analysis**: จัดกลุ่มสินค้าตามมูลค่าและความสำคัญ
- **Lot/Batch Tracking**: ติดตาม Lot Number ของวัสดุ (สำหรับสินค้าที่ต้องการ Traceability)

### ขั้นตอนการทำงาน
1. รับสินค้าเข้าคลังจาก GR (จาก PO Module)
2. ตรวจสอบคุณภาพและบันทึกเข้า Location ในคลัง
3. เมื่อโครงการต้องการวัสดุ สร้าง Material Requisition
4. อนุมัติและเบิกจ่าย (Goods Issue) ตัดสต๊อก
5. กรณีโอนย้ายระหว่างคลัง บันทึก Stock Transfer
6. ตรวจนับสต๊อกประจำงวด (Physical Inventory)
7. ปรับปรุงส่วนต่าง (Adjustment) และสอบสวนสาเหตุ
8. ตรวจสอบ Reorder Point และสร้าง PR สำหรับวัสดุที่ใกล้หมด

### ตัวอย่างข้อมูล
- **Item Code**: MAT-001 (ปูนซีเมนต์ ตราเสือ)
- **Current Stock**: 150 ถุง
- **Warehouse**: คลังหลัก โครงการคอนโดลาดพร้าว
- **Reorder Point**: 100 ถุง
- **Reorder Quantity**: 500 ถุง
- **Last Issue**: 13 มกราคม 2026 (50 ถุง ไปโครงการชั้น 5)
- **Next Physical Count**: 31 มกราคม 2026

### FAQ

**Q:** ระบบรองรับการจัดการคลังหลายแห่งหรือไม่?  
**A:** รองรับครับ สามารถสร้าง Warehouse หลายแห่ง และ Location หลายจุดภายใน Warehouse แต่ละ Item สามารถมีสต๊อกกระจายอยู่ใน Location ต่างๆ ระบบจะรวมยอดแสดงใน Dashboard

**Q:** การตั้ง Reorder Point ควรพิจารณาอะไรบ้าง?  
**A:** ควรพิจารณา (1) ความต้องการใช้เฉลี่ย (Average Consumption), (2) Lead Time ของการสั่งซื้อ, (3) Safety Stock สำหรับรองรับความต้องการเพิ่มขึ้น, (4) ต้นทุนในการเก็บรักษา ระบบสามารถคำนวณแนะนำ Reorder Point ตามสูตร EOQ (Economic Order Quantity)

**Q:** สามารถเบิกจ่ายวัสดุย้อนหลังได้หรือไม่?  
**A:** ได้ครับ แต่ต้องมีสิทธิ์พิเศษ และระบบจะบันทึก Log การเปลี่ยนแปลง เพื่อให้สามารถ Audit ได้ การเบิกจ่ายย้อนหลังจะส่งผลกระทบต่อรายงานสต๊อกและต้นทุนของงวดก่อนหน้า

**Q:** ระบบรองรับการตรวจนับแบบ Cycle Count หรือไม่?  
**A:** รองรับครับ สามารถตั้งค่าให้ตรวจนับบางรายการ (ตามหมวด หรือ ABC Classification) โดยไม่ต้องปิดคลัง ช่วยลดผลกระทบต่อการปฏิบัติงานและตรวจสอบความถูกต้องของสต๊อกได้บ่อยขึ้น

## 7. Fixed Assets (FA)

### คำอธิบาย
โมดูลสำหรับจัดการสินทรัพย์ถาวร เช่น อาคาร รถยนต์ เครื่องจักร เครื่องมือก่อสร้าง ตั้งแต่การซื้อ การคำนวณค่าเสื่อมราคา การบำรุงรักษา จนถึงการจำหน่าย

### ฟีเจอร์หลัก

- **Depreciation Calculation**: คำนวณค่าเสื่อมราคาตามวิธีต่างๆ (Straight-Line, Declining Balance, Units of Production)
- **Asset Transfer**: โอนย้ายสินทรัพย์ระหว่างแผนก, โครงการ, สถานที่
- **Asset Maintenance**: จัดการการซ่อมบำรุง PM (Preventive Maintenance) และ CM (Corrective Maintenance)
- **Asset Disposal**: บันทึกการจำหน่าย ขาย ชำรุด คำนวณกำไร/ขาดทุน
- **Asset Tagging**: สร้าง Barcode/QR Code สำหรับติดสินทรัพย์
- **Insurance Tracking**: ติดตามกรมธรรม์ประกันและวันหมดอายุ

### ขั้นตอนการทำงาน
1. ซื้อสินทรัพย์และบันทึกใน Asset Register
2. กำหนดข้อมูลการคิดค่าเสื่อม (วิธี, อายุการใช้งาน, มูลค่าซาก)
3. ระบบคำนวณค่าเสื่อมราคารายเดือนอัตโนมัติ
4. สร้างแผนการบำรุงรักษา (Maintenance Schedule)
5. บันทึกการซ่อมบำรุงเมื่อถึงกำหนดหรือเกิดปัญหา
6. กรณีโอนย้ายสินทรัพย์ บันทึก Asset Transfer
7. เมื่อสินทรัพย์หมดอายุ/ชำรุด บันทึก Asset Disposal
8. สร้างรายงานสินทรัพย์และค่าเสื่อมสำหรับบัญชีและภาษี

## 8. General Ledger (GL)

### คำอธิบาย
โมดูลบัญชีแยกประเภททั่วไป เป็นหัวใจของระบบบัญชี รับรายการบัญชีจากทุกโมดูล (AP, AR, IC, FA, Payroll) และสร้างงบการเงิน

### ฟีเจอร์หลัก
- **Chart of Accounts (COA)**: ผังบัญชีตามมาตรฐานการบัญชี (Assets, Liabilities, Equity, Revenue, Expense)
- **Journal Entry**: บันทึกรายการบัญชีด้วยตนเอง (Manual Entry)
- **Automated Posting**: รับรายการอัตโนมัติจากโมดูลอื่น
- **Cost Center Accounting**: จัดสรรต้นทุนตามโครงการและแผนก
- **Financial Reports**: งบดุล, งบกำไรขาดทุน, งบกระแสเงินสด, งบแสดงการเปลี่ยนแปลงส่วนของเจ้าของ
- **Trial Balance**: งบทดลอง ตรวจสอบความสมดุล
- **Budget vs Actual**: เปรียบเทียบงบประมาณกับผลการดำเนินงานจริง
- **Period Close**: ปิดงวดบัญชีรายเดือน/รายปี

### ขั้นตอนการทำงาน
1. ตั้งค่า Chart of Accounts ตามโครงสร้างบัญชีของบริษัท
2. รับรายการบัญชีอัตโนมัติจากโมดูลต่างๆ (AP Invoice → Expense, AR Invoice → Revenue)
3. บันทึก Manual Journal Entry กรณีรายการพิเศษ (Accrual, Provision, Adjustment)
4. ตรวจสอบ Trial Balance ว่าสมดุล (Debit = Credit)
5. ทำ Journal Allocation จัดสรรต้นทุนตาม Cost Center
6. สร้างรายงานทางการเงิน (งบดุล, งบกำไรขาดทุน)
7. ปิดงวดบัญชี (Period Close) เพื่อล็อครายการ
8. นำงบการเงินไปวิเคราะห์และจัดทำรายงานต่อผู้บริหาร

- **Account Type**: Asset
- **Balance**: 5,250,000 บาท (Debit)

- **Account Code**: 5110 (Material Cost)

### FAQ

**Q:** Chart of Accounts (COA) คืออะไร?  
**A:** คือผังบัญชีที่แสดงรายการบัญชีทั้งหมดที่ใช้ในองค์กร จัดเป็นหมวดหมู่ตามประเภท (สินทรัพย์, หนี้สิน, ส่วนของเจ้าของ, รายได้, ค่าใช้จ่าย) มีรหัสบัญชี (Account Code) เพื่อความเป็นระเบียบ

**Q:** Manual Journal Entry ใช้กรณีไหน?  
**A:** ใช้กรณีรายการที่ไม่ได้เกิดจากโมดูลอื่นอัตโนมัติ เช่น (1) Accrual รายได้/ค่าใช้จ่ายค้างรับ/ค้างจ่าย, (2) Provision สำรองหนี้สงสัยจะสูญ, (3) Adjustment ปรับปรุงบัญชีท้ายงวด, (4) Allocation จัดสรรต้นทุนร่วม

**Q:** Period Close คืออะไรและทำไมต้องทำ?  
**A:** คือการปิดงวดบัญชี เพื่อล็อครายการไม่ให้แก้ไขย้อนหลัง ป้องกันการเปลี่ยนแปลงข้อมูลหลังจากออกงบการเงินแล้ว ควรทำหลังจากตรวจสอบความถูกต้องและ Reconcile บัญชีทุกรายการแล้ว

**Q:** ระบบสามารถจัดสรรต้นทุนตาม Cost Center อัตโนมัติได้หรือไม่?  
**A:** ได้ครับ สามารถตั้งค่า Allocation Rule เช่น จัดสรรค่าไฟฟ้าตามพื้นที่ใช้สอย, จัดสรรค่าแรงตามชั่วโมงทำงาน ระบบจะคำนวณและสร้าง Journal Entry อัตโนมัติตามสูตรที่กำหนด

## 9. Accounts Receivable (AR)

### คำอธิบาย
โมดูลสำหรับจัดการลูกหนี้การค้า ตั้งแต่การออก Invoice การรับชำระเงิน การติดตามหนี้ จนถึงการตัดหนี้สูญ

### ฟีเจอร์หลัก
- **Invoice Generation**: สร้าง Tax Invoice ตาม Progress Billing หรือ Milestone
- **Payment Receipt**: บันทึกการรับชำระเงิน (โอน, เช็ค, เงินสด)
- **Payment Allocation**: จัดสรรยอดชำระกับ Invoice (Partial, Full, Advance)
- **Aging Report**: รายงานอายุหนี้ลูกค้า (Current, 1-30, 31-60, 61-90, >90 วัน)
- **Reminder & Collection**: แจ้งเตือนและติดตามหนี้ค้างชำระ
- **Credit Note**: ออกใบลดหนี้กรณีส่วนลด หรือคืนสินค้า
- **Bad Debt Write-off**: ตัดหนี้สูญกรณีเรียกเก็บไม่ได้

### ขั้นตอนการทำงาน
1. โครงการเสร็จตาม Milestone หรือ Progress
2. สร้าง Invoice ตามสัญญา (เช่น เสร็จ 30% เรียกเก็บ 30%)
3. ส่ง Invoice ให้ลูกค้าพร้อมเอกสารประกอบ
4. ลูกค้าโอนเงินเข้าบัญชี หรือส่งเช็ค
5. บันทึก Payment Receipt และ Allocate กับ Invoice
6. กรณีลูกค้าชำระเกิน บันทึกเป็น Advance Payment
7. ติดตามหนี้ค้างชำระตาม Aging Report

## 11. Project Management (PM)

### ฟีเจอร์หลัก
- **Project Setup**: สร้างโครงการ กำหนด Start-End Date, Budget, Milestone
- **Work Breakdown Structure (WBS)**: แบ่งงานเป็นลำดับขั้น (Phase → Activity → Task)

- **Resource Management**: จัดสรรแรงงาน เครื่องจักร วัสดุให้งาน
- **Progress Tracking**: บันทึกความคืบหน้างานจริง เปรียบเทียบกับแผน
- **Budget Control**: ติดตาม Budget vs Actual Cost
- **Change Order**: จัดการคำสั่งเปลี่ยนแปลงงาน (Scope Change)
- **Document Management**: จัดเก็บแบบแปลน สัญญา RFI Drawing

### ขั้นตอนการทำงาน
1. สร้างโครงการจาก Opportunity หรือ Bidding ที่ Win

4. กำหนด Dependency ระหว่างงาน (Finish-to-Start, Start-to-Start)
5. จัดสรรทรัพยากร (คน, เครื่องจักร, วัสดุ)
6. สร้าง Baseline Schedule (แผนอ้างอิง)
7. บันทึกความคืบหน้าจริงเป็นประจำ (Daily/Weekly)
8. เปรียบเทียบ Plan vs Actual, Budget vs Cost
9. จัดการ Issue, Risk, Change Order
10. ปิดโครงการและสรุปบทเรียน (Lesson Learned)

### FAQ

**Q:** Gantt Chart คืออะไรและใช้ประโยชน์อย่างไร?  
**A:** คือแผนภูมิแท่งแสดงแผนงานตามแกนเวลา แสดงงานแต่ละอย่าง ระยะเวลา และความสัมพันธ์ (Dependency) ช่วยให้เห็นภาพรวมของโครงการ งานที่ทำพร้อมกันได้ และ Critical Path (งานที่ห้ามล่าช้า)

**Q:** CPI และ SPI คืออะไร?  
**A:** CPI (Cost Performance Index) = Earned Value / Actual Cost, วัดประสิทธิภาพการใช้งบ ถ้า CPI > 1 ใช้น้อยกว่างบ, CPI < 1 เกินงบ / SPI (Schedule Performance Index) = Earned Value / Planned Value, วัดความคืบหน้า ถ้า SPI > 1 เร็วกว่าแผน, SPI < 1 ล่าช้า

**Q:** ระบบรองรับการจัดการ Change Order หรือไม่?  
**A:** รองรับครับ เมื่อลูกค้าขอเปลี่ยนแปลงงาน (Scope Change) สามารถสร้าง Change Order บันทึกรายละเอียดการเปลี่ยนแปลง ผลกระทบต่องบและเวลา และผ่านการอนุมัติ เมื่ออนุมัติแล้วจะ Update ข้อมูล Contract, WBS, Budget อัตโนมัติ

**Q:** สามารถจัดการเอกสารโครงการในระบบได้หรือไม่?  
**A:** ได้ครับ มี Document Management ภายในโมดูล PM จัดเก็บเอกสารแยกตามประเภท (Contract, Drawing, RFI, Submittal, Change Order) รองรับ Version Control และสามารถแชร์ให้ Team Member ดาวน์โหลดได้

## 13. Alert

### คำอธิบาย
โมดูลสำหรับตั้งค่าและจัดการการแจ้งเตือนอัตโนมัติ เมื่อเกิดเหตุการณ์สำคัญหรือตรงตามเงื่อนไขที่กำหนด

### ฟีเจอร์หลัก
- **Event-Based Alert**: แจ้งเตือนเมื่อเกิดเหตุการณ์ (PO Approved, Invoice Overdue)
- **Threshold-Based Alert**: แจ้งเตือนเมื่อค่าเกินหรือต่ำกว่าที่กำหนด (Stock < Reorder Point)
- **Time-Based Alert**: แจ้งเตือนตามกำหนดเวลา (Payment Due in 3 days)
- **Multi-Channel Notification**: ส่งผ่าน Email, SMS, LINE, In-App Notification
- **Alert Routing**: ส่งแจ้งเตือนถึงบุคคลหรือกลุ่มที่เกี่ยวข้อง
- **Alert Dashboard**: แสดง Alert ทั้งหมด สถานะ และการดำเนินการ
- **Escalation**: ส่งต่อหากไม่มีการตอบกลับภายในเวลาที่กำหนด

### ขั้นตอนการทำงาน
1. กำหนด Alert Rule ในระบบ (เงื่อนไข, ผู้รับแจ้ง, ช่องทาง)
2. ระบบ Monitor เหตุการณ์และข้อมูลตาม Rule
3. เมื่อตรงเงื่อนไข ระบบสร้าง Alert และส่งแจ้งเตือน
4. ผู้รับแจ้งดู Alert และดำเนินการ
5. บันทึกการตอบกลับหรือดำเนินการใน Alert Log
6. หากไม่ตอบกลับภายในเวลาที่กำหนด ระบบ Escalate ให้ผู้บังคับบัญชา
7. ปิด Alert เมื่อดำเนินการเสร็จสิ้น

### ตัวอย่างการแจ้งเตือน
- **Stock Alert**: "วัสดุปูนซีเมนต์ (MAT-001) เหลือ 80 ถุง ต่ำกว่า Reorder Point 100 ถุง"

- **Budget Alert**: "โครงการคอนโดลาดพร้าว ใช้งบไปแล้ว 85% (40,800,000 / 48,000,000 บาท)"

### FAQ

**Q:** สามารถปิดการแจ้งเตือนบางประเภทได้หรือไม่?  
**A:** ได้ครับ แต่ละ User สามารถตั้งค่า Alert Preference ได้ เช่น ไม่ต้องการแจ้งเตือน Stock Alert, หรือต้องการแจ้งเตือนเฉพาะผ่าน Email ไม่ต้องการ SMS

**A:** รองรับครับ สามารถผูก LINE Official Account และส่งแจ้งเตือนผ่าน LINE Notify หรือ Messaging API ได้

**Q:** สามารถกำหนดให้แจ้งเตือนหลายคนพร้อมกันได้หรือไม่?  
**A:** ได้ครับ สามารถกำหนดเป็น User Group เช่น "แผนกจัดซื้อ", "ผู้จัดการโครงการ" ระบบจะส่งแจ้งเตือนให้ทุกคนใน Group พร้อมกัน

**Q:** Escalation คืออะไร?  
**A:** คือการส่งต่อ Alert ให้ผู้ที่มีอำนาจสูงขึ้น หากผู้รับแจ้งไม่ตอบกลับภายในเวลาที่กำหนด เช่น แจ้ง User หาก 2 ชั่วโมงไม่ตอบ Escalate ไปหา Manager, หาก 4 ชั่วโมงไม่ตอบ Escalate ไปหา Director

## 14. Sending Email

### คำอธิบาย
โมดูลสำหรับจัดการการส่งอีเมลอัตโนมัติและด้วยตนเอง รองรับการส่งเอกสาร การแจ้งเตือน และการสื่อสารกับลูกค้า/ผู้ขาย

### ฟีเจอร์หลัก
- **Email Template**: สร้างเทมเพลตอีเมลสำหรับเอกสารต่างๆ (PO, Invoice, Quotation)
- **Automated Email**: ส่งอีเมลอัตโนมัติเมื่อเกิดเหตุการณ์ (PO Approved → ส่งให้ Vendor)
- **Bulk Email**: ส่งอีเมลหลายฉบับพร้อมกัน
- **Email Tracking**: ติดตามว่าผู้รับเปิดอ่านหรือยัง
- **Attachment Management**: แนบเอกสาร PDF อัตโนมัติ
- **Email Log**: บันทึกประวัติการส่งอีเมลทั้งหมด
- **SMTP Configuration**: ตั้งค่าเชื่อมต่อ Email Server

### ขั้นตอนการทำงาน
1. ตั้งค่า SMTP Server (Gmail, Office365, หรือ Mail Server เอง)
2. สร้าง Email Template สำหรับเอกสารต่างๆ
3. กำหนด Automated Email Rule (เช่น PO Approved → Send to Vendor)
4. เมื่อเกิดเหตุการณ์ ระบบสร้างอีเมลจาก Template
5. แนบเอกสาร (PDF) อัตโนมัติ
6. ส่งอีเมลและบันทึก Log
7. Track Email Status (Sent, Delivered, Opened, Failed)
8. กรณี Failed ระบบ Retry อัตโนมัติ

### ตัวอย่างการใช้งาน
- **PO to Vendor**: เมื่ออนุมัติ PO ระบบส่งอีเมลพร้อม PDF ให้ผู้ขายอัตโนมัติ
- **Invoice to Customer**: เมื่อสร้าง Invoice ระบบส่งอีเมลพร้อม PDF ให้ลูกค้า
- **Payment Reminder**: ส่งอีเมลเตือนลูกค้าก่อนครบกำหนดชำระเงิน 3 วัน
- **Quotation Follow-up**: ส่งอีเมลติดตามผลหลังส่ง Quotation 7 วัน

### FAQ

**Q:** สามารถใช้ Gmail หรือ Office365 ส่งอีเมลได้หรือไม่?  
**A:** ได้ครับ ตั้งค่า SMTP และ App Password (Gmail) หรือ OAuth (Office365) แต่แนะนำให้ใช้ Email Server เฉพาะ หรือ Email Service (SendGrid, Amazon SES) สำหรับการส่งอีเมลจำนวนมาก

**Q:** ระบบรองรับภาษาไทยในอีเมลหรือไม่?  
**A:** รองรับครับ รองรับ UTF-8 encoding ทั้งหัวข้อ เนื้อหา และชื่อไฟล์แนบเป็นภาษาไทยได้

**Q:** สามารถกำหนดให้ CC หรือ BCC ได้หรือไม่?  
**A:** ได้ครับ สามารถกำหนด CC, BCC ใน Email Template หรือ Email Rule เช่น ส่ง Invoice ให้ลูกค้า CC Manager, BCC Accounting

**Q:** หากส่งอีเมลไม่สำเร็จจะทำอย่างไร?  
**A:** ระบบจะ Retry อัตโนมัติ 3 ครั้ง หาก Failed ทั้งหมดจะแจ้งเตือน Admin และบันทึกใน Failed Email Queue เพื่อส่งใหม่ภายหลัง

## Best Practices สำหรับธุรกิจก่อสร้าง

### การจัดการโครงการ

#### ติดตามความคืบหน้าอย่างสม่ำเสมอ
- อัพเดท Progress รายวันหรือรายสัปดาห์
- ใช้ Gantt Chart และ Earned Value Management (EVM)
- จัดประชุม Project Review ทุกสัปดาห์
- ระบุปัญหาและแก้ไขทันที ไม่ปล่อยให้สะสม

#### จัดการการเปลี่ยนแปลง (Change Management)
- ทุก Change Request ต้องผ่านการอนุมัติ
- ประเมินผลกระทบต่องบและเวลา
- บันทึก Change Order อย่างเป็นทางการ
- Update Baseline Schedule และ Budget

### การจัดการทางการเงิน

#### ควบคุมกระแสเงินสด (Cash Flow)
- คาดการณ์ Cash Flow ล่วงหน้า 3-6 เดือน
- ติดตาม AR Collection ให้ได้ตามแผน
- จัดการ AP Payment ให้สมดุลกับ Cash In
- เก็บ Reserve Fund สำหรับเหตุฉุกเฉิน

#### วิเคราะห์งบประมาณและต้นทุน
- เปรียบเทียบ Budget vs Actual รายเดือน
- คำนวณ Cost Performance Index (CPI) และ Schedule Performance Index (SPI)
- ระบุ Cost Variance และหาสาเหตุ
- ปรับแผนการใช้งบหากจำเป็น

### การจัดการสินค้าคงคลัง

#### ควบคุมสต๊อกให้เหมาะสม
- ตั้ง Reorder Point ที่เหมาะสมตาม Lead Time และ Consumption Rate
- จัดกลุ่มสินค้าตาม ABC Analysis (A = มูลค่าสูง ต้องควบคุมเข้มงวด)
- ตรวจนับสต๊อกประจำงวด (Physical Inventory)
- ป้องกันการสูญหายและลดของเสีย

#### Just-In-Time Delivery
- ประสานงานกับผู้ขายให้ส่งวัสดุตามความต้องการ
- ลด Holding Cost จากการเก็บสต๊อกมากเกินไป
- ใช้ Purchase Requisition วางแผนการซื้อล่วงหน้า

### การจัดการความเสี่ยง

#### ระบุและประเมินความเสี่ยง
- จัดทำ Risk Register แต่ละโครงการ
- ประเมิน Probability และ Impact
- กำหนด Risk Mitigation Plan
- Review ความเสี่ยงใหม่เป็นระยะ

#### ประกันภัยและสัญญา
- ทำประกันโครงการ (Contractor All Risk, Third Party Liability)
- อ่านสัญญาอย่างละเอียด เข้าใจเงื่อนไขและ Penalty
- เก็บเอกสารครบถ้วน กันข้อพิพาทในอนาคต

### การจัดการทีมงาน

#### มอบหมายงานชัดเจน
- กำหนด Roles & Responsibilities อย่างชัดเจน
- มอบหมายงานตามความเชี่ยวชาญ
- ใช้ Approval Workflow ป้องกันความผิดพลาด

#### การสื่อสารที่มีประสิทธิภาพ
- ประชุมสั้นๆ แต่บ่อย (Daily Stand-up)
- ใช้ระบบจัดการเอกสารกลาง (Document Management)
- แจ้งเตือนอัตโนมัติสำหรับงานสำคัญ

### การใช้เทคโนโลยี

#### Digital Transformation
- ใช้ระบบ ERP รวมทุก Module
- ใช้ Mobile App สำหรับงานในสนาม
- เชื่อมต่อ Bank อัตโนมัติ
- ใช้ AI วิเคราะห์ข้อมูลและคาดการณ์

#### Data-Driven Decision
- สร้าง Dashboard แสดงข้อมูลสำคัญ Real-time
- วิเคราะห์ Historical Data หา Pattern
- ใช้ข้อมูลในการตัดสินใจ ไม่ใช้ Feeling

## FAQ ทั่วไป

**Q:** ระบบต้องใช้งานบนคอมพิวเตอร์หรือสามารถใช้บน Mobile ได้?  
**A:** ใช้ได้ทั้งสองแบบครับ มี Web Application สำหรับคอมพิวเตอร์และ Mobile App สำหรับ iOS/Android รองรับการทำงานในสนามและนอกสถานที่

**Q:** ข้อมูลในระบบปลอดภัยหรือไม่?  
**A:** ปลอดภัยครับ มีการเข้ารหัสข้อมูล (Encryption), Backup อัตโนมัติทุกวัน, User Authentication, Role-Based Access Control และ Audit Log ครบถ้วน

**Q:** สามารถ Integrate กับระบบอื่นได้หรือไม่?  
**A:** ได้ครับ มี API (RESTful) สำหรับเชื่อมต่อกับระบบภายนอก เช่น ระบบธนาคาร, ระบบภาษี, ระบบ HR, Third-party Accounting Software

**Q:** มี Training และ Support หรือไม่?  
**A:** มีครับ มี Training ทั้งแบบ On-site และ Online, User Manual ภาษาไทย, Video Tutorial และ Help Desk สำหรับตอบคำถาม

**Q:** สามารถ Customize ระบบตามความต้องการได้หรือไม่?  
**A:** ได้ครับ ระบบออกแบบให้ Flexible สามารถปรับแต่งได้ทั้ง Field, Report, Workflow และสามารถพัฒนา Custom Module ได้หากมีความต้องการพิเศษ

**Q:** ระบบรองรับมาตรฐานการบัญชีไทยหรือไม่?  
**A:** รองรับครับ ระบบพัฒนาตามมาตรฐานการบัญชีไทย (TFRS), รองรับการออกเอกสารภาษีตาม พ.ร.บ.ภาษีมูลค่าเพิ่ม (VAT), ภาษีหัก ณ ที่จ่าย และรายงานภาษีต่างๆ

**Q:** หาก License หมดอายุ ข้อมูลจะหายหรือไม่?  
**A:** ไม่หายครับ ข้อมูลยังคงอยู่ แต่จะไม่สามารถเข้าถึงได้จนกว่าจะต่ออายุ License สามารถ Export ข้อมูลออกมาเป็น Backup ก่อน License หมดได้

**Q:** ระบบมี Limitation จำนวน User หรือธุรกรรมหรือไม่?  
**A:** ขึ้นกับ License ที่ซื้อครับ มี Package ตั้งแต่ SME (10 Users) ไปจนถึง Enterprise (Unlimited) และไม่จำกัดจำนวนธุรกรรม แต่ Database Size จำกัดตาม Package
